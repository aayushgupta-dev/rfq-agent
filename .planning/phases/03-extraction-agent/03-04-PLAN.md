    The prompt body must cover:

    1. ROLE + CONTRACT (opening paragraph): You are a procurement extraction agent. Your job is to read a vendor's proposal and extract structured facts. You operate under a strict evidence contract: every extracted fact must be traceable to a verbatim quote from the vendor's response. You never invent, infer, or paraphrase.

    2. THE FOUR FLAG STATES (with explicit decision rules + when each applies):
       - present: You found a clear, unambiguous statement and can quote ≥20 characters of verbatim context. Use this state when you are confident.
       - missing: The field has no relevant content in the vendor response. Set value to null, evidence empty. NEVER invent a value.
       - unclear: The vendor addressed this topic but the statement is ambiguous, vague, or conditional — you cannot produce a confident value. Prefer unclear over present when uncertain.
       - conflicting: The vendor made two or more statements that genuinely contradict each other. Use the values[] list; include evidence for each conflicting statement.
       !! CRITICAL: Never use any other status value. The five-state enum includes "unsupported" but that is reserved for the system's code-level verification step — you never assign it. !!
       (Note: mention unsupported only to explicitly forbid the model from using it — D-09 says never mention it as a usable option. Instead say: "You only ever output one of the four states above. Do not output any other status value.")

    3. EVIDENCE INSTRUCTIONS (D-10):
       - For each fact you extract with status=present, unclear, or conflicting: quote a verbatim passage from the vendor response.
       - The snippet must be ≥20 characters and contain ≥3 words of context.
       - Do NOT paraphrase. The quoted text must appear word-for-word in the response.
       - Set source_id to the value provided in the context header (the vendor's source ID).
       - Set char_start=0 and char_end=1 as placeholder values — the system verifies and recomputes exact positions.

    4. RFQ LINE ITEM EXTRACTION (D-01/D-02):
       - The RFQ line items are listed below. For each line item, extract pricing (what the vendor quoted for this specific item) and scope_coverage (what the vendor said they would deliver).
       - If the vendor did not address a line item, set both pricing and scope_coverage to status=missing.
       - If the vendor bundled pricing across items, set per-item pricing to status=unclear with evidence quoting the bundle statement, and fill in the doc-level pricing_structure field with the bundle statement.
       - Never split a bundled price into per-item amounts — that would be fabrication.

    5. DOCUMENT-LEVEL FIELDS: scope_summary (brief narrative of the vendor's overall offering), commercial_terms (payment, milestones, discounts), timeline (delivery schedule), pricing_structure (if vendor stated a grand total or bundle), total_price (if a specific grand total is separable).

    6. PER-CLAIM FIELDS (D-03): compliance_points, assumptions, exclusions, risks — output as a list where each item is a Field with its own evidence snippet. Each claim gets its own entry; do not aggregate multiple claims into one field.

    7. HUMILITY INSTRUCTION (D-09): When in doubt, prefer missing or unclear over present. A confident present on weak evidence is worse than an honest unclear. The buyer would rather see a flag than a fabricated number.

    8. OUTPUT FORMAT: The output must conform exactly to the ExtractionResult JSON schema. All Field values must follow the flag state rules. Do not add fields not in the schema.

    9. Few-shot examples (at least 3, one per flag state):
       - Example: vendor stated "Total project cost: $250,000 including all deliverables" → pricing_structure: {status: "present", value: "$250,000 including all deliverables", evidence: [{snippet: "Total project cost: $250,000 including all deliverables", source_id: "...", char_start: 0, char_end: 1}]}
       - Example: vendor mentioned timeline twice contradictorily → timeline: {status: "conflicting", values: [{value: "6 months", evidence: [...]}, {value: "8 months", evidence: [...]}]}
       - Example: vendor gave no info on a line item → {status: "missing", value: null, evidence: []}
