## Spacing Scale

Standard 8-point scale via Tailwind utilities (`p-`, `m-`, `gap-`). Map tokens
to the CSS classes listed; do not hardcode raw pixel values.

| Token | Value | Tailwind class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `p-1` / `gap-1` | Icon gaps, badge inner padding, evidence-snippet gutter |
| sm | 8px | `p-2` / `gap-2` | Compact pill spacing, flag badge padding |
| md | 16px | `p-4` / `gap-4` | Default element spacing, card body padding |
| lg | 24px | `p-6` / `gap-6` | Section padding inside cards and panels |
| xl | 32px | `p-8` / `gap-8` | Between major card groups on a screen |
| 2xl | 48px | `py-12` / `gap-12` | Between screen sections (e.g. summary → full body) |
| 3xl | 64px | `py-16` | Top/bottom page chrome padding |

**Exceptions:**

- Stage rail (D-01) nav items: 44px minimum touch target height (`min-h-11`) — accessibility requirement, not a spacing token.
- Evidence snippet pill (D-07): inner padding `px-2 py-1` (8px/4px) — asymmetric to keep quotes compact.
- Flag badge (D-08): inner padding `px-1.5 py-0.5` (6px/2px) — deliberate density for inline use.
- Streaming progress bar (D-25): full-width, no horizontal padding.

## Typography

System `font-sans` throughout. Four roles, two weights.

| Role | Size | Tailwind | Weight | Weight name | Line Height | Usage |
|------|------|----------|--------|-------------|-------------|-------|
| Body | 14px | `text-sm` | 400 | regular | 1.5 (`leading-relaxed`) | Default prose, evidence snippets, field values |
| Label | 12px | `text-xs` | 500 | medium | 1.4 (`leading-snug`) | Flag badges, stage rail items, table headers, metadata |
| Heading | 20px | `text-xl` | 600 | semibold | 1.2 (`leading-tight`) | Screen section headings, panel titles |
| Display | 28px | `text-3xl` | 600 | semibold | 1.1 (`leading-tight`) | Page-level screen titles (one per page) |

**Rules:**
- Two weights only: 400 (regular) and 600 (semibold). Do not use 700/bold anywhere.
- `text-muted-foreground` for secondary text (evidence labels, timestamps, metadata).
- `text-foreground` for primary text.
- Never use `text-base` (16px) — it is between body (14px) and heading (20px) and creates ambiguity.

## Color

Sourced from `globals.css` CSS variables. Use semantic token names, not raw oklch values.

| Role | CSS Token | Light-mode approx | Usage |
|------|-----------|-------------------|-------|
| Dominant (60%) | `--background` | white (`oklch(1 0 0)`) | Page background, main content area |
| Secondary (30%) | `--card` / `--sidebar` | near-white (`oklch(0.985 0 0)`) | Stage rail, cards, panels, category sections |
| Accent (10%) | `--primary` | near-black (`oklch(0.205 0 0)`) | Reserved list below |
| Muted | `--muted` / `--muted-foreground` | light grey | Evidence snippets background, empty-state copy |
| Destructive | `--destructive` | red (`oklch(0.577 0.245 27.325)`) | Destructive actions only (see below) |
| Border | `--border` | `oklch(0.922 0 0)` | All dividers, card outlines |

**Accent (`--primary`) reserved for — explicit list:**
- Active stage rail item indicator (D-01)
- Primary CTA buttons ("Run Extraction", "Run Comparison", "Load Sample", "Regenerate RFQ")
- "comparable" badge fill on the comparability matrix (D-11) — one color signals "all clear"
- Highlighted evidence span underline in the drill-down source passage (D-07)

**Accent is NOT used for:** nav hover states, card borders, section headings, body text emphasis.

**Flag state badge palette** (semantic, separate from the 60/30/10 surface palette):

| FlagStatus | Badge bg token | Badge text | Notes |
|------------|---------------|------------|-------|
| `present` | `bg-green-100` | `text-green-800` | Tailwind semantic; not a shadcn token |
| `missing` | `bg-red-100` | `text-red-800` | High-visibility — absence is first-class (§8) |
| `unclear` | `bg-amber-100` | `text-amber-800` | Caution |
| `conflicting` | `bg-orange-100` | `text-orange-800` | Between unclear and missing in severity |
| `unsupported` | `bg-slate-100` | `text-slate-600` | Evidence span could not be verified |

**Comparability badge palette** (Comparison screen matrix, D-11):

| ComparabilityVerdict | Badge | Notes |
|----------------------|-------|-------|
| `comparable` | accent bg (`bg-primary text-primary-foreground`) | |
| `partially` | `bg-amber-100 text-amber-800` | |
| `not_comparable` | `bg-red-100 text-red-800` | |

**Destructive** (`--destructive`): used only on file-remove ("Remove vendor") confirmation. No other destructive actions in this phase.

```
<html>
  <body>
    <div class="flex min-h-screen">
      <!-- Stage rail: fixed left, 240px wide (desktop), collapses to icon bar (mobile) -->
      <nav class="w-60 shrink-0 ...">  <!-- sidebar tokens -->
      <!-- Main content area -->
      <main class="flex-1 min-w-0 overflow-y-auto">
```

Stage rail width: 240px desktop / 60px icon-only on `< md` breakpoint.
Main content max-width: `max-w-5xl mx-auto` (desktop centered, full-width on narrow).

### Route structure (App Router, `app/(buyer)/`)

```
app/
  layout.tsx              ← root layout (html/body shell only)
  (buyer)/
    layout.tsx            ← buyer shell: stage rail + main content wrapper
    rfq/page.tsx          ← UI-01 RFQ Overview (D-21, D-22)
    input/page.tsx        ← UI-02 Vendor Input (D-04..D-06)
    extraction/page.tsx   ← UI-03 Extraction Review (D-07..D-10)

    trace/page.tsx        ← UI-05 Prompt Trace / Prompt Pack (D-15)
  page.tsx                ← redirect to /rfq (or replace with redirect())
```

### Responsive breakpoints (D-26 — reflow, not redesign)

| Breakpoint | Tailwind | Stage rail | Content |
|------------|----------|------------|---------|
| < 768px (`sm`) | default | Hidden; hamburger opens drawer | Full-width, single-column |

## Screen-by-Screen Interaction Contracts

### UI-01: RFQ Overview (`/rfq`) — D-21, D-22

**Information hierarchy:**
1. Summary panel (top): event name, dates, 8 line items listed, key commercial expectations — `Card` with `lg` padding.
2. Full structured body below: scope → timelines → line items (each as a sub-card) → commercial expectations → vendor questionnaire → compliance — grouped by category with `Separator` between groups.

**Interactions:**
- On load: render committed `data/rfq.json` immediately (no loading state).
- "Regenerate RFQ" button (top-right, `variant="outline"`): calls `GET /data/rfq`, shows `Skeleton` in the summary panel while loading, replaces content on response.
- No destructive actions on this screen.

**Empty state:** Not applicable — RFQ is always present (committed `data/rfq.json` is the fallback).

---

### UI-02: Vendor Input (`/input`) — D-04..D-06

**Information hierarchy:**
1. One-click sample load (hero, D-04): three `Card` tiles (Thorough / Cheap / Fluff vendor), each with a "Load Sample" primary `Button`. Immediate — no spinner needed (data is local JSON).
2. Paste path (secondary): `Textarea` (8 rows) + `Input` for vendor name + "Submit for Extraction" `Button`. Label above each field.
3. File upload path (tertiary, D-05): drag-target zone or `<input type="file">` accepting `.pdf,.docx,.xlsx,.pptx`. Shows filename after selection + "Extract & Analyze" `Button`. Weak-extraction `Alert` appears if backend returns empty/minimal text.

**Interactions:**
- Sample load → immediately navigates to `/extraction` with the selected vendor loaded into session cache (D-02). No SSE on this screen.
- Paste submit / file upload → navigates to `/extraction` while SSE runs (loading state lives there, D-25).
- Vendor name `Input` is required before submit (client-side validation, native `required` attribute).
- "Remove" on a loaded vendor: `Button variant="ghost"` with destructive icon — no confirmation dialog (not a data-loss action; user can re-load trivially).
- Weak-extraction `Alert` (D-05): variant `"destructive"` (amber/warning tone), copy: "Text extraction returned very little content — paste the vendor response manually for best results."

**Empty state (no vendors loaded yet):** Shown above the paste/upload sections — `"No vendors loaded yet. Load a sample above or paste a response below."` in `text-muted-foreground`.

---

### UI-03: Extraction Review (`/extraction`) — D-07..D-10

**Information hierarchy (D-08, UI-06 buyer-first):**
1. Vendor selector: `Tabs` component — one tab per loaded vendor (D-10).
2. Gaps & Risks panel (always-visible, D-08): `Card` with `Alert`-style header listing every field with a non-`present` FlagStatus. Color-coded by severity (`missing` → red, `conflicting` → orange, `unclear` → amber, `unsupported` → slate). Count in panel title: "3 gaps, 1 conflict".
3. By-category sections (D-09): each schema category (`scope`, `pricing`, `commercial_terms`, `timeline`, `compliance`, `assumptions`, `exclusions`, `risks`) as a `Card`. Fields listed as rows: field label | flag `Badge` | value (or "—" if absent) | evidence snippet (always visible inline under value, `text-xs text-muted-foreground`, D-07).
4. Evidence drill-down (`Collapsible` per field, D-07): expands to show the source passage with the cited span underlined in accent color.

**Streaming state (D-25):**
- On navigation from `/input` with a pending extraction: `Progress` bar at top of main content, status text from SSE `status` events ("Aligning to RFQ… Extracting fields… Grounding evidence…").
- Fields populate progressively via SSE `partial` events if the backend emits them; otherwise show `Skeleton` rows until `result` event.
- On `error` event: `Alert variant="destructive"` with `ErrorPayload.message` + "Try reloading the vendor or check the AI service." — never a blank screen (D-25, §8).

**Interactions:**
- Tab switch: instant (session-cached result, D-02).
- Evidence `Collapsible` trigger: chevron icon button, `aria-expanded` toggle.
- No destructive actions.

**Empty state (no extraction yet):** "Select or load a vendor on the Input screen to begin extraction." with a link to `/input`.

---

### UI-04: Vendor Comparison (`/comparison`) — D-11..D-14

**Information hierarchy (D-11, UI-06 buyer-first):**
1. Clarifications & Attention panel (always-visible, D-12): `Card` listing every `AttentionPoint` + generated clarification questions. Each item links to its dimension/field via an anchor or highlight. Panel title: "N items need attention".
2. Comparability matrix (hero, D-11): vendors as columns, 6 dimensions as rows. Each cell: `Badge` (comparable / partially / not_comparable) + one-line reason. Sticky header row (vendor names) on horizontal scroll.
3. "Comparability determined in code from evidence — not a model verdict" note (D-14): `text-xs text-muted-foreground` below the matrix, always visible.
4. 8-line-item offer table (drill-down, `Collapsible` D-11): vendors × line items, showing offer value + FlagStatus badge per cell.
5. Per-dimension narrative (drill-down, `Collapsible`): model-generated summary per dimension with grounded evidence citations.

**Stable order (D-13):** Vendors always rendered in input order (no sort controls). Readiness X/N count rendered as `"Data readiness: 4/6 dimensions comparable"` — labelled explicitly, not a score.

**Streaming state (D-25):** Same pattern as Extraction — `Progress` + status text + `Skeleton` → populated on `result`.

**Error state (D-25):** `Alert variant="destructive"` — "Comparison could not complete. {message}" — never blank.

**Empty state (no extractions loaded):** "Run extraction on ≥1 vendor on the Extraction screen before comparing." with link.

---

### UI-05: Prompt Trace / Prompt Pack (`/trace`) — D-15

**Information hierarchy:**
1. Trace selector `Tabs`: one tab per committed trace file (`docs/traces/*.json`). Default: the comparison trace (most narrative-rich for the demo, D-19).
2. Per-trace view:
   - Pipeline stages as a vertical timeline: Input → Prompt (id + version) → Raw model output → Grounded/clamped final output.
   - Downgrade diff section (D-15): side-by-side or annotated view of `ClampEntry[]` — model-proposed vs code-clamped verdict, **highlighted** (amber background on changed cells). Labelled: "Code overruled the model on N verdicts".
   - Raw model output in `<pre>` block with monospace font (`font-mono text-xs`), inside `ScrollArea`.
3. Prompt Pack list (below trace, `Tabs` or `Collapsible`): one entry per prompt ID from `services/ai/prompts/registry.py`. Each entry shows: prompt id, version, one-line intent (from frontmatter), link to `docs/prompts/` doc. Not a code editor — read-only display.

**Interactions:** Static render from committed JSON — no streaming, no SSE on this screen. `ScrollArea` for long raw outputs. Copy-to-clipboard button on raw output block (native `navigator.clipboard`).

**Empty state:** Not applicable — committed traces are always present.

## Interaction & State Contracts

### SSE streaming pattern (D-23, D-25)

All streaming initiates on navigation to `/extraction` or `/comparison` when a pending request exists in session. Steps:

1. `Progress` component at 0%, status text: "Starting…"
2. Each `status` event: update status text + advance progress estimate.
3. Each `partial` event (if any): incrementally populate fields.
4. `result` event: render full output, hide `Progress`.
5. `done` event: confirm complete (progress to 100%, then hide bar after 500ms).
6. `error` event: show `Alert variant="destructive"` with `ErrorPayload.message`. Progress bar hidden. Retry not automatic.

Session cache (D-02): store `ExtractionResult` / `ComparisonResult` in React Context or `sessionStorage`. Re-visiting the screen renders from cache instantly — no re-run.

### Focus and keyboard nav

- Stage rail nav items: `<a>` elements, keyboard-navigable.
- `Tabs` components: Radix handles arrow key nav.
- `Collapsible` triggers: `<button>` with `aria-expanded`.
- Evidence drill-down: focus moves into expanded content.
- Minimum touch target for all interactive elements: 44px height (via `min-h-11`).

### Loading / skeleton treatment

- Extraction / Comparison: `Skeleton` rows in each category card while streaming. Skeleton height matches approximate final content height to avoid layout shift.
- RFQ regeneration: `Skeleton` in summary `Card` only; full body remains visible.
- File upload: browser native progress is sufficient; no custom upload bar.
