## API Contract (verified against app.py, extraction.py:109-114, extraction.py:225-231, comparison.py:896-897)

This is the ground-truth contract. Every plan that touches extraction or comparison MUST use
these exact field names and understand the payload asymmetry.

### POST /extract/vendor
  Request body: { vendor_response: VendorResponse, rfq: RFQ }
  ← NOT { vendor, rfq }. The key is "vendor_response". FastAPI validates against ExtractionRequest
    which has field `vendor_response: VendorResponse`. Sending "vendor" → 422 Unprocessable Entity.

### POST /compare/vendors
  Request body: { extractions: ExtractionResult[], rfq: RFQ }
  ← "extractions" key (list of ExtractionResult). Confirmed in ComparisonRequest.
  Minimum 2 vendors enforced server-side (_MIN_VENDORS=2) — sending 1 → 422.

### SSE event: "status"
  payload: { message: string, phase: string }
  ← phase is the natural progress signal. Extraction phases: "model" then "grounding".
  ← Do NOT use +20% counter; drive StreamProgress off the known phase sequence.
  Extraction phase sequence: "model" → "grounding" → done
  Map to progress values: "model" → 40%, "grounding" → 80%, done → 100%

### SSE event: "result" — ASYMMETRIC between extraction and comparison
  Extraction result payload (extraction.py:225-231):
    { ...ExtractionResult fields spread at top level..., downgrade_report: DowngradeReport }
    ← NOT a bare ExtractionResult. The ExtractionResult fields are at the TOP LEVEL alongside
      a sibling "downgrade_report" key. This is intentional (ponytail comment in extraction.py).
    ← To get a clean ExtractionResult safe to re-POST to /compare/vendors, the client MUST
      normalize: destructure downgrade_report out before caching, store only the ExtractionResult fields.
    ← The DowngradeReport structure: { entries: DowngradeEntry[] }

  Comparison result payload (comparison.py:896-897):
    { ...ComparisonResult fields... }
    ← bare ComparisonResult — no extra siblings. Cast directly.

### SSE event: "error"
  payload: { code: string, message: string, recoverable: boolean }

### SSE event: "done"
  payload: {}

### Normalization in api.ts (REQUIRED)
  When an extraction "result" event arrives, separate downgrade_report before caching:
    const { downgrade_report, ...extractionResult } = event.payload as any
    setExtraction(vendorName, extractionResult as ExtractionResult)
    // keep downgrade_report separately if the Extraction screen needs it for display
  What is stored in extractions[vendorName] is a clean ExtractionResult — safe to re-POST
  to /compare/vendors as extractions[].
